# japanese-technical-writing

日本語の技術文書を作成、確認、修正する際の方針を定めるスキル。

## 適用対象

このスキルは主に次の2つの場面で使う。

1. ソースコードや技術情報を調査した後に、説明資料を新規作成する。
2. 既存の技術資料を校正し、構成・文法・体裁を整える。

## 全体の方針

技術文書の品質は以下の優先順位で決まる。後から修正するコストが高いものを先に確認すること。

1. テンプレート方針: どの体裁・章立てを採用するかを最初に確定する。
2. 構成・論理: 説明順序、前提、理屈のつながりを確認する。
3. 文法・文意: 主語、係り受け、指示語の一意性を確認する。
4. 体裁・表現: 文体、表記、冗長語句を確認する。

分かりやすさのために正確さを犠牲にしてはならない。
作業を行う際は、関連資料を読み、作業対象の文章と文脈を正確に理解すること。

## 読者の内部モデル構築を支援する原則

この skill が生成する文書は、読者が頭の中に対象の内部モデルを構築できることを重視する。
事実の羅列ではなく、読者が対象の構造と因果を把握し、説明された状況以外にも応用できる理解を得られるように設計すること。

以下の原則は、テンプレートの節構成や説明順序の根拠となるものであり、すべてのテンプレートに共通して適用する。

### 情報の提示順序

読者の内部モデルは以下の順に立ち上がる。各段階は前の段階に依存するため、この順序で情報を提示すること。

1. 境界: 対象は何であり、何ではないか。似た概念との違いは何か。
2. 構成要素と役割: 主要な部品は何か。それぞれ何を担当しているか。
3. 因果の流れ: 最も単純なケースで、入力から出力まで何がどう変わるか。
4. 詳細と例外: 中心構造が把握できてから、例外条件や詳細仕様を扱う。

境界が見えないと構成要素の判別ができない。構成要素が見えないと因果の流れを追えない。
この依存関係があるため、順序を崩すと読者は情報を断片として処理せざるを得なくなり、理解コストが上がる。

### 因果の明示

「何があるか」だけでなく「なぜそうなるか」を示すこと。構造の説明には因果関係を含めること。

因果が明示されると、読者は説明された具体的状況を超えて、対象の挙動を予測できるようになる。
因果が省略されていると、読者は個々の事実を個別に記憶するしかなくなり、応用が難しくなる。

### 用語の導入方法

新しい用語や固有名詞を導入する際は、名前の前に役割を示すこと。

- 避けるべき順序: 「X とは ○○ である」（名前 → 役割）
- 推奨する順序: 「○○ を担う仕組みを X と呼ぶ」（役割 → 名前）

読者は役割が分かっていれば、名前をモデル上のラベルとして記憶できる。
役割が不明なまま名前だけ導入すると、読者にとって接続先のない記号になり、記憶に残りにくい。

## テンプレート方針（第0段階）

執筆または改稿を開始する前に、`references/00-template-policy.md` を参照し、次を必ず確定すること。

- `template_mode`: `inherit` / `user_template` / `skill_default`
- `template_type`: `skill_default` の場合、テンプレートカタログから選択したテンプレート ID
- `must_keep`: 維持必須の章立て、見出し、順序、表記規則
- `can_change`: 再編可能な範囲

テンプレートの優先順位は以下とする。

1. ユーザーの明示指示
2. 元資料・既存文書の制約
3. スキル標準テンプレート（目的別）

`skill_default` モードの場合は、文書の目的に応じてテンプレートカタログから適切なテンプレートを選択する。
テンプレートカタログと各テンプレートの詳細は
`references/00-template-policy.md` および `references/templates/` を参照すること。

選択可能なテンプレートは以下の5種類である。

- `explanation`: 技術説明書 — 仕組み・概念を読者に理解させる
- `investigation-report`: 調査報告書 — 調査の結果と考察を報告する
- `design-doc`: 設計文書 — 設計判断とその根拠を記録する
- `howto`: 手順書 — 手順を正確に伝え、読者が再現できるようにする
- `incident-report`: 障害報告書 — 障害の経緯・原因・再発防止策を記録する

### 第0段階の指摘範囲

第0段階の指摘は、テンプレートの選択と確定に関するものに限定する。具体的には以下を扱う。

- テンプレートモードの判定（`inherit` / `user_template` / `skill_default`）
- テンプレートの選択・変更の提案（例: 「この文書は `explanation` テンプレートに切り替えるべきである」）
- `must_keep` と `can_change` の確定

章立ての順序、節の依存関係、概念の導入順序といった構成上の問題は、第0段階ではなく第1段階（構成・論理）で扱う。
第0段階でテンプレート変更を提案する際に構成上の根拠に触れる場合は、
簡潔に理由を述べるにとどめ、詳細な構成分析は第1段階で行うこと。

## 文章の確認と修正

確認作業は以下の順序で行うこと。前の段階で問題が見つかった場合は、それを解決してから次に進むこと。

### 第1段階: 構成・論理の確認（`references/01-structure.md` 参照）

以下の問いに答える形で評価すること。「問題なし」と即断せず、各問いに対して根拠を持って答えること。

- この文書、または節が伝えようとしていることを一文で言えるか。
- 読み手がまだ知らない概念に依存した説明が、その概念の説明より前に登場していないか。
- 各節・段落の主張と、その根拠・結論の関係は明確か。論理の飛躍や前提の欠落がないか。
- 同じ内容が複数箇所に分散していないか。矛盾している記述がないか。

### 第2段階: 文法・文意の確認（`references/02-grammar.md` 参照）

第1段階の確認が完了してから行うこと。

### 第3段階: 体裁・表現の確認（`references/03-style.md` 参照）

第2段階の確認が完了してから行うこと。冗長語句・推量表現・文体の統一などをまとめて確認する。
体裁だけを先に確認することはしない。

## 確認結果の返し方

- テンプレート方針（第0段階）の確定結果を確認結果の冒頭に示すこと。以下の4項目を明記する。
  - `template_mode`: 判定した値（`inherit` / `user_template` / `skill_default`）と判定理由
  - `template_type`: `skill_default` の場合は選択したテンプレート ID。それ以外の場合は「該当なし」と記載し、参考として最も近いテンプレート ID を併記する。
  - `must_keep`: 維持必須の要素。校正時は、既存文書の主題・技術的内容を維持必須とし、章立て・見出し・表現は `can_change` に含める。ユーザーが維持を指示した要素があればここに追加する。
  - `can_change`: 再編可能な範囲を具体的に列挙する。
- 方針に対する指摘がある場合は、4項目の直後にまとめて述べる。
- 各指摘は `指摘事項`、`修正案`、`理由` の順で返すこと。修正案が不要な場合は省略してよい。
- 指摘は段階順（テンプレート方針→構成→文法→体裁）で示すこと。
- 同一の問題が複数の段階に該当する場合は、最初に該当する段階で指摘し、後の段階では「第N段階 指摘ID参照」の形式で参照にとどめること。
- 編集を依頼された場合は指示に従って文章を修正すること。必要に応じて、修正点とその理由を簡潔に説明すること。

## 文章の作成

- 第0段階で確定したテンプレート方針に従って文章を作成する。
- ユーザーの指定がない場合は、`references/00-template-policy.md` のテンプレートカタログから文書の目的に適したテンプレートを選択し、対応する `references/templates/` 内のテンプレートに従って作成する。
- テンプレート方針の確定結果（`template_mode`、`template_type`、`must_keep`、`can_change`）は、文書本文には含めない。作成前にユーザーへ提示するか、作成結果の報告時に付記する。ユーザーとの対話が不可能な場合は、作成結果の報告時に確定結果を添えること。
- 文章は常体（である調）で作成すること。ただし、ユーザーが敬体（ですます調）を指定した場合はそれに従うこと。
- 事実と推測を混同しない。不確かな内容は条件や根拠を明示すること。
