# japanese-doc-review

日本語ドキュメントの問題を見つけ、改善判断を支援するためのスキル。
レビュー対象には、README、設計メモ、仕様説明、運用手順、調査報告を含む。

## 使う場面

具体的なレビュー対象本文に対するレビュー要求が明示されている場合に使う。

レビュー対象本文とは、次のいずれかである。

- ユーザーが提示した本文
- 既存ファイルに書かれた本文
- エージェントが作成済みのドラフト本文

具体的なレビュー対象本文がない場合は、次の形式で返す。

```markdown
レビュー対象本文が必要です。
対象本文、対象ファイル、または作成済みドラフトを提示してください。
```

## レビュー結果の形式

レビュー対象本文がある場合は、`references/review-result-format.md` を読む。
返答は同資料の固定形式に従う。

レビュー結果は、後続作業のコンテキストとして使われる。
対象範囲、規定事項、確認事項、今回のレビュー範囲、指摘の表記は固定形式に従って書く。
レビュー観点の見出しは、`構成・論理`、`文意・文法`、`体裁・表現` の3つを常にこの順序で出す。
単一観点指定の場合も、レビュー対象外の観点を省略しない。
レビュー対象外の観点では、`今回のレビュー範囲: 含まない`、該当する `理由`、`指摘: - 今回はレビュー対象外。` を書く。

## 用語と定義

- `規定事項`: 本文の正誤、矛盾、明確さ、重要度を判定するために正として扱う定義、制約、目的、対象範囲、判断基準、または本文から必然的に導ける重要事実。
- `確認事項`: 本文だけでは規定事項または指摘の要否を確定できないときに出す質問。回答によって規定事項が確定するか、指摘の要否または重要度が変わる場合に使う。
- `対象`: 指摘が対応する本文中の具体的な位置と引用。ファイルパス、行番号、見出し、段落、箇条書き項目と、本文中の短い引用を組み合わせて示す。
- `根拠`: 指摘を対象または規定事項から導く理由。対象のどの表現が、どの誤解、矛盾、判断ミス、理解コストを生むかを説明する。
- `今回のレビュー範囲`: 各レビュー観点を今回のレビューに含めるかを示す項目。値は `含む` または `含まない` のどちらかにする。

## 対象範囲

`対象範囲` には、今回レビューした実在本文の範囲を書く。
文書全体をレビューした場合は `全文` と書く。
文書の一部をレビューした場合は、ファイルパス、見出し、節名、行番号から判別できる範囲を書く。

レビュー対象本文の代わりに示された構成案、要約、執筆予定、未作成部分は対象範囲に含めない。
対象範囲外をレビュー済みの範囲として扱わない。

## 規定事項

`規定事項` はレビュー全体で共通のセクションとして出す。
規定事項には、レビュー判断を拘束する命題だけを書く。
規定事項は、後続の指摘または確認事項の判定に直接使うものだけを書く。
指摘や確認事項に接続しない情報は、正しい情報であっても規定事項に書かない。
観点選択、重要度判定、出力形式など、このスキル内部のルールを適用した結果は規定事項に書かない。
それらは各観点の `今回のレビュー範囲`、`理由`、`重要度` に書く。
ファイル調査、コマンド実行、外部資料確認で得た観測結果は、原則として規定事項に書かない。
観測結果は、本文の具体的な問題を示すために必要な場合だけ、該当指摘の `根拠` に書く。
規定事項は、後続の複数の判断を拘束する命題、または書かないと指摘の成立条件が不明になる命題だけに絞る。
一度だけ使う観測結果、指摘文中で説明すれば足りる根拠、本文の要約は規定事項に採録しない。

規定事項にしない例:

- 依頼語から導いた観点選択の結果。
- ファイル調査やコマンド実行で得た個別の観測結果。
- 対象文書の中心テーマや構成の要約。

規定事項に含める情報:

- ユーザーが、本文に対して正として扱うよう明示した定義、制約、目的、対象範囲、判断基準
- レビュー対象本文に明示された定義、制約、目的、対象範囲
- 本文の記述から論理的に必然といえ、後続の指摘または確認事項の判定に直接使う重要事実
- 確認事項への回答で、本文に対して正として扱うことになった定義、制約、目的、対象範囲、判断基準、重要事実
- ユーザーが今回の依頼コンテキストとして提示した過去レビュー結果の `規定事項`

規定事項には、推測、一般知識からの補完、条件が未確定の情報を含めない。
規定事項は、本文要約や調査結果一覧として使わない。

1項目には、1つの定義、制約、目的、対象範囲、判断基準だけを書く。
規定事項が多くなる場合は、同じ制約や目的をまとめ、後続の指摘判断に必要な形へ圧縮する。

規定事項と本文が矛盾する場合は、規定事項を正として扱い、本文側を構成・論理の `高` 重要度の指摘にする。
規定事項同士が矛盾する場合や、提示された情報だけでは規定事項として採用すべき命題を判断できない場合は、構成・論理の `高` 重要度の指摘に加えて確認事項にも出す。
矛盾がある場合も、本文と規定事項から判断できる範囲でレビューを進める。

## 確認事項

`確認事項` はレビュー全体で共通のセクションとして出す。
確認事項は最大3件に絞る。
確認事項がある場合も、本文と規定事項から判断できる範囲でレビューを進める。

確認事項で扱うもの:

- 未定義の用語:
  本文中で説明なしに使われており、既知として扱うかどうかで指摘の要否が変わる用語。
- 未説明の構成・制約:
  本文中で正として扱われており、正しいかどうかで記述の妥当性が変わるシステム構成、権限、運用制約、仕様上の制限。
- 判断基準:
  本文中の評価や選択理由を理解するために必要だが、本文だけでは確定できない基準。

確認事項は、指摘を保留するための質問にしない。
本文だけで矛盾、基準不足、定義不足を指摘できる場合は指摘として出し、修正時に確定が必要な点だけを確認事項にする。
好みや文体、広い読者像、レビュー開始前の必須確認としての質問は確認事項に含めない。

## 指摘の選び方

指摘は、レビュー対象範囲の全体に照らして選ぶ。
本文の冒頭から見つけた順に、指摘枠を埋めない。

長い仕様書、指示書、手順書、報告書をレビューする場合は、先に全体を走査して次の情報を把握してから指摘を選ぶ。

- 文書の目的、読者、対象範囲、対象外、中心概念
- 実行、判断、承認、運用、判定に使う条件と順序
- 同じ概念、条件、状態、手順が複数箇所に出ている位置
- 未定義の用語、未説明の判断基準、例外条件、外部資料への依存

長文では、個別の表現よりも、離れた箇所の食い違い、手順の抜け、判断基準の不足を優先する。
複数箇所をまたぐ問題は、原因ごとに1つの指摘にまとめる。
まとめた指摘の `対象` には、問題を再現できる代表箇所を2つまで示し、近い見出し、段落、箇条書き位置、短い引用を組み合わせる。
ファイルではない貼り付け本文でも、見出しや節名を使って読者が本文中の位置を探せるようにする。

確認事項に回すのは、回答によって規定事項、指摘の要否、重要度、修正方針が変わる点だけにする。
本文だけで矛盾、基準不足、定義不足を示せる場合は、確認事項だけにせず指摘として出す。
未決事項や本文中の質問は、そのまま転記せず、レビュー判断に影響する最大3件へ圧縮する。

優先する指摘:

- 文書の目的、対象範囲、中心概念、判断基準に関わる矛盾
- 複数箇所に分散した重複説明、古い説明、条件の食い違い
- 実行、判断、運用、判定の成否を本文だけで再現できない基準不足
- 前段で実行や判断を求めているが、必要な基準や定義が後段まで出てこない構成
- 上流観点の高・中重要度の問題があるため、下流観点の指摘を出すと修正順序を誤らせる問題

同じ原因から出ている問題は、個別に並べずまとめて指摘する。
まとめた指摘では、代表的な対象を1つ以上示す。

## レビュー観点

レビュー観点は次の3つに固定する。

- 構成・論理:
  文書全体の主張、説明順序、規定事項、因果、重複、矛盾を扱う。
- 文意・文法:
  文単位の主語、係り受け、指示語、文意の曖昧さを扱う。
- 体裁・表現:
  冗長表現、推量表現、表記揺れ、文体統一を扱う。

依頼語が次のいずれかに当てはまる場合は、観点指定として扱う。

- `レビューして`、`問題点を指摘して`、`セルフレビューして`: 観点指定なし。
- `校正して`: `文意・文法` と `体裁・表現` の複数観点指定。
- `文法だけ`、`文意だけ`、`主語`、`係り受け`、`指示語`: `文意・文法` の単一観点指定。
- `誤字脱字だけ`、`表記揺れだけ`、`文体だけ`、`冗長表現だけ`: `体裁・表現` の単一観点指定。
- `全部`、`総合`、`全観点`: 全観点指定。

以降、観点指定と観点の依存関係により今回指摘を出す対象になった観点を、レビュー対象の観点と呼ぶ。

レビュー対象の観点は、次の規則で決める。

- 観点指定なしの場合は、`構成・論理` だけをレビュー対象の観点にする。
- 単一観点指定の場合は、指定された観点だけをレビュー対象の観点にする。
- 複数観点指定または全観点指定の場合は、観点の依存関係を守る。
- 上流観点に `高` または `中` の指摘がある場合、指定された下流観点はレビュー対象の観点にしない。
- 上流観点に `高` または `中` の指摘がない場合だけ、指定された次の下流観点をレビュー対象の観点にする。

レビュー対象の観点に応じて、次の補助資料を読む。

- `構成・論理`: `references/01-structure.md`
- `文意・文法`: `references/02-grammar.md`
- `体裁・表現`: `references/03-style.md`

`理由` には短い定型文を書く。

- 指定なしのため、最も上流の観点を優先。
- ユーザー指定の観点。
- ユーザー指定の観点外。
- 構成・論理のレビューを優先。
- 指定観点だが、構成・論理に高・中重要度の指摘があるため。
- 指定観点だが、文意・文法に高・中重要度の指摘があるため。

## 重要度

重要度は、レビュー対象の観点における影響で判定する。
重要度は `高`、`中`、`低` の3段階で示す。

- 高: 読者が内容を誤解する、手順を誤る、判断を誤る可能性が高い問題。構成・論理の欠落、規定事項不足、矛盾、技術的に危険な曖昧さを含む。
- 中: 理解コストが明確に上がる問題。説明順序の不自然さ、根拠不足、係り受けの曖昧さ、用語の導入不足を含む。
- 低: 理解は可能だが、読みやすさ、一貫性、信頼感を下げる問題。冗長表現、文体の揺れ、表記揺れ、軽微な誤字を含む。

## 指摘の基準

指摘は、具体的な対象と根拠を示せるものに限る。
印象、好み、推測は、指摘の根拠として扱わない。

指摘は、レビュー対象の観点ごとに最大5件に絞る。
5件は出力上限であり、その観点の問題が5件以下であることを意味しない。
5件を超える場合は、その観点で重要度が高い問題から優先して出す。
同じ原因の問題が複数ある場合はまとめてよいが、代表的な対象を1つ以上示す。

`対象` には、位置と引用を含める。
ファイルをレビューする場合は、可能な範囲でファイルパスと行番号を含める。
本文が貼り付けで提供され、行番号がない場合は、最も近い見出し、段落、箇条書き位置を示す。
文書全体の構成問題では、問題が発生している構造位置と代表的な引用を示す。

`根拠` は、対象に含めた引用、または規定事項に列挙した命題に接続する。
根拠では、対象の表現がどの誤解、矛盾、判断ミス、理解コストを生むかを説明する。

`修正方針` には、修正の判断基準または作業方針を書く。
必要な追加情報、削る断定、並べ替える情報、用語定義の追加位置を示す。

指摘を反映した本文の修正、再レビュー、次観点のレビューは、レビュー結果を受け取った後続工程として扱う。
