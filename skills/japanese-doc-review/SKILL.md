# japanese-doc-review

日本語ドキュメントの問題を見つけ、改善判断を支援するためのスキル。
レビュー対象には、README、設計メモ、仕様説明、運用手順、調査報告などを含む。

## 使う場面

このスキルは、依頼にレビュー要求が明示されている場合に使う。

レビュー要求とは、文章の問題点を見つけ、その結果を指摘または出力前の改善に使うことを求める依頼である。
具体的には、次のいずれかである。

- ユーザーが提示した文章のレビュー、校正、読みやすさの確認、問題点の指摘を求めている。
- エージェントがこれから出力する文章について、提出前のセルフレビュー、見直し、品質確認を求めている。
- 第三者視点や別視点で文章を確認することを求めている。

## 用語と定義

- `前提`: レビュー対象本文に対して正として扱う事実または合意済み定義。本文が前提と矛盾する場合は、本文側を問題として扱う。前提は参考情報ではなく、本文の正誤や明確さを判定する基準である。
- `確認事項`: 本文だけでは前提または指摘の要否を確定できないときに出す質問。回答によって前提が確定するか、指摘の要否が変わる場合に使う。
- `対象`: 指摘が対応する本文中の具体的な記述単位。見出し、段落、文、箇条書き項目、短い引用のいずれかで示す。複数箇所にまたがる問題は、同じ問題を確認できる代表的な記述単位を1つ以上示す。
- `根拠`: 指摘を本文または前提から導く理由。対象に示した記述単位のどの表現が、なぜ問題を生むかを説明する。
- `未確認の観点`: 優先順位の都合で今回は扱わない観点。上流観点に高・中重要度の問題がある場合、より細かい観点は未確認としてよい。

## レビュー結果の扱い

レビュー結果を返答に含めるか、提出前の改善に使うかを依頼内容から判断する。

### 返答に含めない場合

「セルフレビューしてから出して」「見直してから回答して」のように、ユーザーが回答や文章の提出前に見直しを求めている場合に使う。
レビュー結果を返答に含めない場合は、レビューを出力前の品質確認として扱う。
ユーザーへの返答には、依頼された回答または文章を、レビューを反映した形で含める。
レビュー結果、指摘一覧、修正理由は含めない。
前提が確定しない内容は、断定しない。
ユーザーへの返答では、条件が分かる形で記述する。
確認事項は返答に含めない。

セルフレビューで本文を書き換える場合は、元の文章にある情報、ユーザーが明示した前提、本文から直接導ける範囲だけを使う。
曖昧表現を具体化するときに、未確認の運用手順、エラー内容、担当者、数値、対象範囲などを補わない。
具体化に必要な情報がない場合は、断定を避け、条件が分かる表現にするか、曖昧な断定を削る。

### 返答に含める場合

「レビューして」「校正して」「問題点を指摘して」「第三者視点で見て」のように、ユーザーが指摘の提示を求めている場合に使う。
レビュー観点と固定形式の指摘を出す。
本文の書き換えは、ユーザーが「直して」「修正版も出して」「この指摘を反映して」と明示した場合に行う。

レビュー結果を返答に含めるか判断できない場合は、レビュー対象の文章がユーザーから提示されているかで判断する。
ユーザーが提示した文章を確認する依頼では、レビュー結果を返答に含める。
エージェントがこれから出す回答や文章の見直しでは、レビュー結果を返答に含めない。

## レビューの優先順位

### レビュー観点

レビューでは、依頼内容に合う最も上流の観点を優先する。

- 構成・論理: 文書全体の主張、説明順序、前提、因果、重複、矛盾を扱う。
  参照: `references/01-structure.md`
- 文意・文法: 文単位の主語、係り受け、指示語、文意の曖昧さを扱う。
  参照: `references/02-grammar.md`
- 体裁・表現: 冗長表現、推量表現、表記揺れ、文体統一を扱う。
  参照: `references/03-style.md`

上流観点に高・中重要度の問題がある場合は、より細かい観点の低重要度の問題を指摘に含めない。
`references/` の観点別ルールは、実際に扱うレビュー観点に対応するものだけを読む。

ユーザーが観点を明示していない場合は、上流の観点を優先して問題を選ぶ。
ユーザーが「誤字脱字だけ」「表記揺れだけ」のように観点を限定した場合は、その範囲に限定する。
表記揺れや文体統一を扱う場合、ユーザーが表記基準を指定していなければ `references/03-style.md` の最小表記基準を使う。
外部の好みや任意のスタイルガイドを、明示なしに根拠として扱わない。

### 重要度

重要度は `高`、`中`、`低` の3段階で示す。

- 高: 読者が内容を誤解する、手順を誤る、判断を誤る可能性が高い問題。構成・論理の欠落、前提不足、矛盾、技術的に危険な曖昧さを含む。
- 中: 理解コストが明確に上がる問題。説明順序の不自然さ、根拠不足、係り受けの曖昧さ、用語の導入不足を含む。
- 低: 理解は可能だが、読みやすさ・一貫性・信頼感を下げる問題。冗長表現、文体の揺れ、表記揺れ、軽微な誤字を含む。

## 指摘の基準

### 採用基準

指摘は、具体的な対象と根拠を示せるものに限る。
印象、好み、推測は、指摘の根拠として扱わない。

レビュー結果を返答に含める場合、指摘は最大5件に絞る。
5件を超える場合は、重要度が高く、上流の問題から優先して出す。
同じ原因の問題が複数ある場合はまとめてよいが、代表的な対象を1つ以上示す。

### 確認事項

レビュー結果を返答に含める場合、本文だけでは断定できないがレビュー品質に影響する情報を `確認事項` として最大3件出してよい。
確認事項は、回答によって前提が確定するか、指摘の要否が変わる質問に限る。
確認事項がある場合も、本文と前提から判断できる範囲でレビューを進める。
確認事項だけでレビューを止めない。

前提に含めてよいもの:

- ユーザーが明示した用語定義
- ユーザーが確認事項に回答して確定した用語、構成、制約
- レビュー対象本文の中で明確に定義され、以後の記述を判定できる情報

前提に含めないもの:

- エージェントの推測
- 書き手の意図の推測
- 「この読者なら知っているはず」という判断
- 具体的な判定基準へ落ちない読者像
- 好みや品質目標

確認事項で扱うもの:

- 未定義の用語:
  本文中で説明なしに使われており、既知として扱うかどうかで指摘の要否が変わる用語。
  例: 「差分同期」は、この文書では既知の用語として扱う前提か。
- 未説明の構成・制約:
  本文中で前提化されており、正しいかどうかで記述の妥当性が変わるシステム構成、権限、運用制約、仕様上の制限。
  例: 「worker は常に1プロセスだけ起動する」という制約は正しい前提か。
- 判断基準:
  本文中の評価や選択理由を理解するために必要だが、本文だけでは確定できない基準。
  例: 「採用理由」で重視しているのは、実装コストより運用時の安全性である、という前提か。

確認事項にしない質問:

- 読者像、文書目的、掲載先などを広く尋ねる質問
- 好みや文体の方向性を尋ねる質問
- レビュー開始前の必須質問として扱うこと

確認事項への回答で確定した情報は、必要に応じて次回以降の `前提` として扱える。

## レビュー結果の形式

レビュー結果を返答に含める場合は、次の形式で返す。

```markdown
レビュー観点: 構成・論理
対象範囲: 全文
未確認の観点: 文意・文法、体裁・表現

確認事項:
- 「差分同期」は本文中で定義されていない。この文書では既知の用語として扱う前提か。

前提:
- 「worker」は非同期処理の実行単位である。
- このシステムでは、worker は同時に1プロセスだけ起動する。

**指摘 1**
重要度: 高
対象: 「...」

問題:
...

根拠:
...

修正方針:
...
```

`確認事項` は、確認事項の条件を満たす質問がある場合に出す。
`前提` は、正として扱える確定情報がある場合に出す。

指摘がない場合は、次の形式で返す。

```markdown
レビュー観点: 構成・論理
対象範囲: 全文
未確認の観点: 文意・文法、体裁・表現

指摘なし。
この観点では、具体的な対象と根拠を示せる問題は見つからなかった。
```
