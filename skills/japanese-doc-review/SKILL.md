# japanese-doc-review

日本語ドキュメントの既存原稿をレビューするためのスキル。
主な対象は技術文書だが、README、設計メモ、仕様説明、運用手順、調査報告などの日本語ドキュメントにも適用できる。

このスキルは文章の新規作成やテンプレート適用を目的としない。
通常の説明、調査報告の作成、メモ作成、README 作成、設計文書の新規作成では使わない。

## 使う場面

このスキルは、ユーザーがレビュー行為を明示した場合だけ使う。

- 「レビューして」
- 「校正して」
- 「読みやすさを見て」
- 「問題点を指摘して」
- 「セルフレビューしてから出して」
- 「見直してから回答して」
- 「第三者視点で見て」

ユーザーがセルフレビューについて何も言わず、単に文章の作成や説明を依頼した場合は、このスキルを使わない。

## 出力モード

レビュー結果の消費先に応じて、出力を切り替える。

### 非表示レビュー

自分の出力品質を上げるためのレビュー。

「セルフレビューしてから出して」「見直してから回答して」のように、主目的が最終本文の提出である場合に使う。
レビュー結果、指摘一覧、修正理由は出さず、改善済み本文だけを出す。
不明な前提がある場合は、ユーザーへの確認事項として出さず、断定を避けるか、本文中で条件を明示する。

### 表示レビュー

レビュー結果をユーザーまたは別のエージェントが読むためのレビュー。

「レビューして」「校正して」「問題点を指摘して」「第三者視点で見て」のように、主目的が指摘の提示である場合に使う。
レビュー観点と固定形式の指摘を出す。
修正文は、ユーザーが「直して」「修正版も出して」「この指摘を反映して」と明示した場合だけ出す。

迷う場合は、レビュー対象の本文をユーザーが渡しているなら表示レビュー、エージェントがこれから書く本文なら非表示レビューとして扱う。

## レビュー観点

レビューでは、依頼内容に合う最も上流の観点を優先する。

- 構成・論理: 文書全体の主張、説明順序、前提、因果、重複、矛盾を扱う。
  参照: `references/01-structure.md`
- 文意・文法: 文単位の主語、係り受け、指示語、文意の曖昧さを扱う。
  参照: `references/02-grammar.md`
- 体裁・表現: 冗長表現、推量表現、表記揺れ、文体統一を扱う。
  参照: `references/03-style.md`

より細かい観点は、上流観点の高・中重要度の問題を押しのけて扱わない。
必要な参照だけを読む。

ユーザーが観点を明示していない場合は、上流の観点を優先して問題を選ぶ。
ユーザーが「誤字脱字だけ」「表記揺れだけ」のように観点を限定した場合は、その範囲に限定する。
ユーザーが上流観点とより細かい観点を同時に指定した場合でも、上流観点に高・中重要度の問題があれば、より細かい観点の低重要度の指摘は出さない。

## 指摘の採用基準

指摘は、具体的な対象と根拠を示せるものだけにする。
印象、好み、推測だけに基づく指摘は出さない。

表示レビューの指摘は最大5件に絞る。
5件を超える場合は、重要度が高く、上流の問題から優先して出す。
同じ原因の問題が複数ある場合はまとめてよいが、代表的な対象を1つ以上示す。

## 重要度

重要度は `高`、`中`、`低` の3段階で示す。

- 高: 読者が内容を誤解する、手順を誤る、判断を誤る可能性が高い問題。構成・論理の欠落、前提不足、矛盾、技術的に危険な曖昧さを含む。
- 中: 理解コストが明確に上がる問題。説明順序の不自然さ、根拠不足、係り受けの曖昧さ、用語の導入不足を含む。
- 低: 理解は可能だが、読みやすさ・一貫性・信頼感を下げる問題。冗長表現、文体の揺れ、表記揺れ、軽微な誤字を含む。

より細かい観点の低重要度の問題は、上流観点の高・中重要度の問題がある場合は出さない。

## 表示レビューの用語

- `対象`: 指摘が対応する本文中の具体的な記述単位。見出し、段落、文、箇条書き項目、短い引用のいずれかで示す。複数箇所にまたがる問題は、同じ問題を確認できる代表的な記述単位を1つ以上示す。
- `根拠`: 指摘を本文または前提から導く理由。対象に示した記述単位のどの表現が、なぜ問題を生むかを説明する。
- `前提`: レビュー時に正として扱う確定情報。ユーザーが明示した定義、確認事項への回答、または本文中で明確に定義された情報だけを含める。本文が前提と矛盾する場合は本文側を問題として扱う。不確かな情報、推測、曖昧な合意、読者像の仮定は含めない。
- `未確認の観点`: 優先順位の都合で今回は扱わない観点。上流観点に高・中重要度の問題がある場合、より細かい観点は未確認としてよい。

## 確認事項

表示レビューでは、本文だけでは断定できないがレビュー品質に影響する前提を `確認事項` として最大3件出してよい。
確認事項だけでレビューを止めない。

確認事項で扱うもの:

- 本文中で未定義のまま使われている専門用語
- 説明なしに前提化されているシステム構成、制約、判断基準
- 指摘として断定するには根拠が足りないが、読み手の理解に影響する前提

確認事項で避けるもの:

- 「読者は初心者ですか、上級者ですか」のような粗いペルソナ質問
- 文書目的や掲載先を毎回聞くこと
- レビューを進める前の必須質問化

確認事項への回答で確定した情報は、必要に応じて次回以降の `前提` として扱える。

## 表示レビューの形式

表示レビューでは、次の形式で返す。

```markdown
レビュー観点: 構成・論理
対象範囲: 全文
未確認の観点: 文意・文法、体裁・表現

確認事項:
- 「差分同期」は本文中で定義されていない。この文書では既知の用語として扱う前提か。

前提:
- 「worker」は非同期処理の実行単位である。

**指摘 1**
重要度: 高
対象: 「...」

問題:
...

根拠:
...

修正方針:
...
```

`確認事項` は必要な場合だけ出す。
`前提` は、正として扱える確定情報がある場合だけ出す。

指摘がない場合は、次の形式で返す。

```markdown
レビュー観点: 構成・論理
対象範囲: 全文
未確認の観点: 文意・文法、体裁・表現

指摘なし。
この観点では、具体的な対象と根拠を示せる問題は見つからなかった。
```
