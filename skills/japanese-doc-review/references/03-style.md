# STYLEの方針

STYLEでは、文章の表現と体裁に関する方針を示す。
誤字、脱字、余字、固有名詞や用語の明らかな誤記は `TYPO` で扱う。

## 太字による強調の禁止

文章中で Markdown 記法の太字(`**`)を使って強調してはならない。
太字は読み手の環境によって表示が異なるため、認識のズレにつながる。
また、太字を使わないと情報の重要度が伝わりにくくなる文章は書くべきではない。

ただし、以下のように節の見出しや区切りを示す印として定型的に使う場合に限り例外的に認める。

**LABEL**

## 表記揺れの検出

同じ概念、同じ作業、同じ資料、同じシステム、同じ組織を指す表現は、文書内で同じ表記に統一すること。
本文に表記ルール、用語集、正式名称、略称の指定がある場合は、それを正として扱う。

表記揺れとして扱うもの:

- 表記ルールと本文中の表記が一致していないもの。
- 同じ概念を指す固有名詞、略称、和名、英名が混在しているもの。
- 同じ資料や項目を指す可能性が高い日本語名と略称が混在し、本文だけでは別物として区別できないもの。

表記揺れを指摘する場合は、揺れている複数の表記と、それぞれの出現位置を `対象` に示す。
本文中に表記ルールがある場合は、その表記ルールを `根拠` に接続する。
同じ概念かどうかが本文だけで確定できない場合は、断定せず「同じ概念を指す場合は統一する」と修正方針に書く。

正式名称や禁止表記に含まれる語が、指標名、画面名、ログ名、資料名などの複合名詞の一部として出ている場合は、
同じ役割、作業、資料、システム、組織を指すと本文から判断できるときだけ表記揺れとして指摘する。
別概念の名称要素である可能性が残る場合は、問題を断定しない。
その語を表記ルールの対象に含めるかどうかが本文だけで決まらない場合は、表記揺れとして断定しない。
指摘として出せる場合の `修正方針` は、正式な指標名や画面名として必要なら表記ルールに例外または正式名称を追加し、
不要なら表記ルールに合う名称へ統一する、という条件付きの方針にする。

## 冗長な語句の削除

以下の語句が含まれている場合、削除して文意が通るかを確認する。

- 「としては」
- 「においては」
- 「に対しては」
- 「については」

文意が通るなら削除する。
文意が通らない場合は、語句を残すのではなく、文の構造を見直して語句なしで意味が通る文に書き直す。
ただし「Aとしては正しいがBとしては誤りである」のように明示的な対比構文で使う場合は残してよい。

## 曖昧な範囲表現の回避

範囲を説明する際に、以下の語句を用いて範囲を曖昧にしないこと。

- 「など」
- 「等」

範囲を説明する際に「AやBなど」と書いた場合、読み手は「など」に何が含まれるかを判断できない。
「など」「等」を使う場合は例を示す際に限定的に使うこと。
列挙が網羅的であれば「など」を外す。
網羅できない場合は、列挙した項目の上位概念で範囲を示すか、条件を明記して範囲を限定する。

## 曖昧な推量表現の回避

以下の推量表現は避けること。

- 「思われる」
- 「考えられる」
- 「可能性がある」

事実であれば断定し、不確かであれば根拠や条件を明示する。

条件下で発生する事象を記述する場合は、推量ではなく条件を明示した表現を使う。

- 避けるべき表現: 「障害が発生する可能性がある」（根拠が不明）
- 許容する表現: 「メモリが不足した場合、障害が発生する」（条件を明示）
- 許容する表現: 「設定値が閾値を超えると、処理が中断される」（条件を明示）

「〜する場合がある」は、発生条件が前文または同文中に明示されている限り許容する。
条件が示されていない「〜する場合がある」は推量表現と同様に扱い、条件を追記するか断定に書き換えること。

## 文体の統一

同一の文書内で文体を混在させないこと。
同じ主体の同じ動作を説明するときは、同じ述語を使うこと。
たとえば「確認します」と「確認いたします」が同じ動作を指す場合は、どちらか一方に統一する。
ユーザーが文体を指定していない場合は、本文中で多く使われている文体、または読み手に余計な敬語関係を生まない簡潔な文体に寄せる。
